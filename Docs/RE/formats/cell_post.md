## 1. Overview and Core Verdict

The `.post` file extension family (specifically `*.ted.post`) represents **baked diffuse lighting post-patches** generated by the offline terrain editor.

- **No Runtime Consumption:** The shipped client binary completely ignores `.post` files at runtime. Cell streaming loops strictly open `.ted` extensions.
- **Development Tool Role:** These files represent full standalone backup copies of edited cells, allowing the developer tools to record raw changes while patching only specific channels (baked diffuse lighting) in the active client files.

---

## 2. On-Disk Layout

Every `.ted.post` file is a fixed-size **46,987-byte (0xB78B)** binary blob. It has no file magic, header markers, or version flags. It uses the exact 5-block terrain cell layout as the runtime `.ted` files:

## 3. Editor Copy-Then-Patch Save Protocol

The offline terrain editor function `Ted_ExportCell_Entry` updates cell heightfield and lighting assets using a copy-then-patch sequence:

1. **Memory Packing:** The editor packs the heightfield grid, calculated normals, texture slots, direction flags, and diffuse lighting values (multiplied by `2.0` on export) into a flat 46,987-byte memory buffer.
2. **Backup Output:** The editor writes this complete memory buffer to `*.ted.post` (creating a standalone backup copy of the cell workspace state).
3. **Live In-Place Patching:** The editor opens the active client terrain file `*.ted` and updates **only the diffuse color channel** (Block 5) by seeking to offset **`30,087`** and writing the final `16,900` bytes. 
   - This in-place patch ensures that baked shadow adjustments and lighting map changes are updated in the live runtime game files without risk of shifting cell vertex geometry (heights/normals), which could break collision alignment.
